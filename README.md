## About

I build software because I like watching something abstract turn into something a real person actually uses — a tutoring business running its schedule off a dashboard I wrote, a classmate using a curriculum I built to learn JavaScript in a language that isn't a translation.

I think in systems before I think in syntax. Before a line of UI gets written, I'm already asking how auth will work, what the data model looks like, and what happens when the happy path fails. That instinct came from building things that other people depend on, not from a course that told me to do it that way.

What I enjoy building sits at a specific intersection: software that teaches (an AI-assisted EdTech platform), software that runs a business (a tutoring dashboard), and software that argues for something I believe in (a renewable-energy site built for a real competition, not a portfolio piece).

The engineer I'm becoming cares more about whether a system is still correct under pressure than whether it looks finished in a screenshot.

## Engineering Philosophy

This is less a list of values and more a description of how I actually work.

**Engineering over coding.**
Writing code that runs is the easy part. The real work is the schema design, the auth flow, the decision about what happens when a request fails halfway through — the parts that don't show up in a demo.

**Quality over quantity.**
I'd rather ship three things I can defend in detail than ten things I'd rather not be asked about. Every one of my active repositories has a visible, checkbox-tracked roadmap — including the parts that aren't done yet.

**Security and structure are not optional extras.**
Token rotation, input validation, and rate limiting go in from the first commit — even on a project nobody but me is reviewing. A monorepo gets split into proper packages before the feature count forces it.

**Build for someone, not for a portfolio.**
The software I respect most is the kind a specific person actually depends on — a business owner, a student, a developer who doesn't read English-first documentation. That's the bar I hold my own work to.
